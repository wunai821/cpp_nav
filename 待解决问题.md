
# 2. 采样空间过小，底盘变“智障” (planner.yaml)
   当前：dwa_linear_samples: 5, lateral: 5, yaw: 5问题：$5 \times 5 \times 5 = 125$ 条轨迹。在 RM 场地上遇到极其刁钻的障碍物（如乱跑的兵或者立柱），125 条轨迹很容易全部处于高 Cost 状态导致机器人“卡死”。修改：既然你是纯 C++ 高性能单进程，算力完全不是问题。建议至少设为 linear: 15, lateral: 15, yaw: 20（生成 4500 条轨迹）。另外：max_vx_mps: 1.0 在 RM 比赛中太慢了（热身赛建图可以，正赛像个活靶子）。一般正赛最高速度要拉到 2.5 - 3.0 m/s 左右。

# 3. 定位容易发生“空间跃迁” (localization.yaml)
当前：max_position_jump_m: 1.2

问题：两帧之间（比如 20Hz，0.05秒内）如果 ICP 匹配导致坐标突变 1.2 米，那绝对是匹配到错误的相似墙面（退化环境）了。这会让你的 TF 树发生剧烈抖动，导致 Planner 直接算出离谱的 cmd_vel 让底盘疯跑。

修改：RM 哨兵底盘极限加速度下，0.05秒最多移动十来厘米。建议将 max_position_jump_m 改为 0.2 或 0.3。超过这个阈值的位姿更新直接作为 Outlier 丢弃，纯靠 IMU+Odom 撑过这一帧。
