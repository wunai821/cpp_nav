2. 并发性能优化：避免在持锁时进行阻塞操作 (Thread Join)
在 CleanupClients() 函数中，现在的逻辑是：

std::lock_guard<std::mutex> lock(clients_mutex_);
// ... 如果客户端断开 ...
if (client->reader_thread.joinable()) {
  client->reader_thread.join(); // 🔴 危险：在这里阻塞等待线程结束
}
风险：CleanupClients() 是在 PublishJson 和 BroadcastText 里面调用的。如果某个读线程 (reader_thread) 正在因为某种原因阻塞（比如正在执行系统调用），join() 就会卡住。这会导致 clients_mutex_ 一直被占用，进而完全卡死其他正在尝试 Publish 数据的线程，影响导航核心算法的实时性。
改进：将需要 join 的线程移出锁的保护范围后再进行 join：
void CleanupClients() {
  std::vector<std::thread> threads_to_join;
  {
    std::lock_guard<std::mutex> lock(clients_mutex_);
    for (auto it = clients_.begin(); it != clients_.end(); ) {
      if (!(*it)->connected.load(std::memory_order_acquire)) {
        if ((*it)->reader_thread.joinable()) {
          // 把线程的所有权转移出来，不在这里 join
          threads_to_join.push_back(std::move((*it)->reader_thread)); 
        }
        it = clients_.erase(it);
      } else {
        ++it;
      }
    }
  } // 释放 clients_mutex_锁
  
  // 在锁外安全地 join
  for (auto& t : threads_to_join) {
    if (t.joinable()) {
      t.join();
    }
  }
}


# v1. 🚨 致命隐患：防御 OOM (内存溢出) 攻击或异常
在 ReadFrame 函数中，你直接将客户端传来的 payload_size 用于 vector 的分配：

C++
// 如果收到恶意包或者粘包导致 payload_size 是一个巨大的数字，比如几个 G
frame->payload.resize(static_cast<std::size_t>(payload_size)); 
风险：这会导致 std::bad_alloc 直接让整个导航进程崩溃（Crash）。
改进：增加一个最大帧大小限制（比如 10MB）：

C++
constexpr std::uint64_t kMaxPayloadSize = 10U * 1024U * 1024U; // 10MB
if (payload_size > kMaxPayloadSize) {
  utils::LogWarn("foxglove", "Payload size exceeds limit");
  return false; 
}
frame->payload.resize(static_cast<std::size_t>(payload_size));
